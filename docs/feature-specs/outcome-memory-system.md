# Feature Specification: Outcome Memory System

## 1. Purpose
The Outcome Memory System (OMS) is the database and logic foundation of the **Career Intelligence Compounding Loop**. It captures the outcomes of every job application submitted through the execution engine (e.g., whether it resulted in a rejection, interview, offer, or ghosting) and pairs it with the initial opportunity scoring prediction.

By comparing predicted success scores against real-world conversion rates, the OMS calculates prediction errors, schedules automated follow-up updates, and feeds clean validation signals back into the ML calibration models.

---

## 2. User Value
Directly operates on the **Outcome Data** layer of the **Career Intelligence Compounding Loop**.
- **Compounding Loop Connection**: Without outcome memory, CareerPilot is a static automation utility. By saving outcome success rates, the system learns what resume positioning, skill profiles, and companies convert. As more users log outcomes, the matching model calibrations grow more accurate, providing better recommendations for everyone.
- **User Benefit**: The platform proactively learns from rejection or interview feedback. Users receive smarter recommendations over time that are tuned to what hiring managers are actively selecting in the current market.

---

## 3. Requirements
- **Outcome Database Schema**: Store application outcomes, status transition histories, target company names, and prediction snapshots.
- **Outcome Recording API**: Expose endpoints to log status updates, copy-paste rejection email text (for LLM analysis), or mark offers.
- **Prediction Error Tracking**: Calculate and store the delta between the initial opportunity score and the final conversion outcome.
- **Follow-up Scheduling Logic**: Automate database triggers or background events to email/nudge the user for status updates if an application has been in `SUBMITTED` state with no updates for 14 days.
- **Outcome Analytics**: Synthesize dashboard metrics representing interview conversion rate, average time-to-first-response, and ghost rate.
- **Calibrator Feed**: Expose clean data interfaces for the ML Platform to retrieve labeled training records.

## 4. Database Changes

### Table: `application_outcomes`
Tracks the chronological result of each application submission alongside prediction metadata.

| Field Name | Data Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | PRIMARY KEY, DEFAULT gen_random_uuid() | Unique identifier. |
| `application_id` | UUID | FOREIGN KEY references `job_applications(id)`, UNIQUE, NOT NULL | Parent job application. |
| `predicted_fit_score` | NUMERIC(5,2) | NOT NULL | Snapshot of `fit_score` generated by Opportunity Intelligence at submission time. |
| `final_outcome` | VARCHAR(50) | NOT NULL | `APPLIED`, `INTERVIEW_SCHEDULED`, `REJECTED`, `OFFERED`, `GHOSTED`, `WITHDRAWN`. |
| `rejection_reason` | TEXT | NULLABLE | Feedback text or notes provided by the company. |
| `rejection_stage` | VARCHAR(50) | NULLABLE | e.g., `RESUME_SCREEN`, `PHONE_SCREEN`, `TECH_TEST`, `FINAL_ROUND`. |
| `prediction_error` | NUMERIC(5,2) | NULLABLE | Numeric delta indicating accuracy of predicted fit. |
| `days_to_response` | INTEGER | NULLABLE | Elapsed days between submission and first response (positive or negative). |
| `follow_up_date` | TIMESTAMPTZ | NOT NULL | Scheduled time to prompt user for status update. |
| `follow_up_completed` | BOOLEAN | NOT NULL, DEFAULT FALSE | State of status update reminders. |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | Record creation time. |
| `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | Last update time. |

### Table: `application_status_history`
Tracks historical status changes for time-to-response calculations.

| Field Name | Data Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | PRIMARY KEY, DEFAULT gen_random_uuid() | Unique identifier. |
| `application_id` | UUID | FOREIGN KEY references `job_applications(id)`, NOT NULL | Parent application. |
| `previous_status` | VARCHAR(50) | NULLABLE | Prior state. |
| `new_status` | VARCHAR(50) | NOT NULL | Current state. |
| `changed_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | Timestamp of change. |

### Indexes
- `idx_app_outcomes_app_id` on `application_outcomes(application_id)`
- `idx_app_outcomes_outcome` on `application_outcomes(final_outcome)`
- `idx_app_outcomes_follow_up` on `application_outcomes(follow_up_date)` WHERE `follow_up_completed` IS FALSE

## 5. API Endpoints

### Record Application Status/Outcome
- **HTTP Method**: `POST`
- **Route**: `/api/v2/applications/{application_id}/outcome`
- **Path Parameters**:
  - `application_id` (string): Application UUID.
- **Request Body (JSON)**:
```json
{
  "new_status": "REJECTED",
  "rejection_reason": "We decided to move forward with candidates whose experience more closely matches...",
  "rejection_stage": "RESUME_SCREEN"
}
```
- **Response Payload (JSON)**:
```json
{
  "application_id": "890ea3c7-1122-3344-5566-778899aabbcc",
  "outcome_id": "012ea3c7-1122-3344-5566-778899aabbcc",
  "final_outcome": "REJECTED",
  "prediction_error": 85.0,
  "updated_at": "2026-06-09T02:04:18Z"
}
```
- **HTTP Status Codes**:
  - `200 OK`: Outcome registered successfully.
  - `401 Unauthorized`: Invalid JWT.
  - `404 Not Found`: Application record not found.

## 7. Services

### Class: `OutcomeMemoryService`
Manages application lifecycle endings, calculates prediction errors, and handles follow-up operations.

- **Method**: `record_status_change`
  - **Inputs**:
    - `application_id` (str): Target application UUID.
    - `status` (str): Next state of the application.
    - `notes` (Optional[str]): Feedback description.
    - `stage` (Optional[str]): Application phase.
  - **Return Type**: `ApplicationOutcomeResponse`
  - **Responsibilities**:
    - Updates `job_applications.current_status`.
    - Logs history in `application_status_history`.
    - Fetches or updates `application_outcomes` record.
    - If status is terminal (e.g., `REJECTED`, `OFFERED`), computes `prediction_error`.
      - **Heuristic**: Assigns numeric success value (e.g., `REJECTED` at resume screening = 0.0, `INTERVIEW` = 50.0, `OFFER` = 100.0) and subtracts from `predicted_fit_score`.
    - Calculates `days_to_response` based on initial application submission timestamp.
    - Disables further automatic follow-up tasks (`follow_up_completed = True`).

- **Method**: `trigger_scheduled_follow_ups`
  - **Inputs**: None.
  - **Return Type**: `int` (count of alerts sent)
  - **Responsibilities**:
    - Queries `application_outcomes` where `follow_up_date <= NOW()` and `follow_up_completed = False`.
    - Iterates over matched records: publishes `notification.schedule` events to prompt user for updates.
    - Sets `follow_up_completed = True` to prevent repeat alerts.

## 9. Background Jobs
- **Follow-up Poller Cron**: Runs every 12 hours (`0 */12 * * *`). Invokes `OutcomeMemoryService.trigger_scheduled_follow_ups` to identify stale applications and nudge candidates.
- **Ghost Application Detection Job**: Runs weekly to auto-transition applications in `SUBMITTED` state for > 45 days to `GHOSTED` status.

---

## 10. Acceptance Criteria
- **Scenario**: User records an interview invitation.
  - **Given**: An application with status `SUBMITTED`.
  - **When**: The user records status change to `INTERVIEW_SCHEDULED`.
  - **Then**: The system logs the history state, computes elapsed days to response, schedules a new follow-up 7 days out, and updates the dashboard view.
- **Scenario**: Application is auto-flagged as ghosted.
  - **Given**: An application was submitted 46 days ago and has no logged updates.
  - **When**: The weekly ghost detection background job runs.
  - **Then**: The system changes the status to `GHOSTED`, sets prediction error, and logs the change.

---

## 11. Edge Cases
- **Negative Latency Status Transitions**: User enters status changes in non-chronological order (e.g., logs an interview, then subsequently uploads the initial application submission details).
  - **Resolution**: The service checks the timestamps of existing status history records. If an older status change is submitted, the history table appends it chronologically, but the primary status on the application table is kept as the highest-advancement state.
- **No Response from Employer**: A company never responds to an application.
  - **Resolution**: The system defaults to auto-marking the application as `GHOSTED` after 45 days. This labels the outcome data correctly to prevent skewing the ML platform with eternally open applications.
