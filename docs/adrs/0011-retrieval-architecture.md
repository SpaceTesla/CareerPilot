# ADR 0011: Hybrid Multi-Stage Retrieval Architecture

## Status
Accepted

## Context
CareerPilot V2 relies on accurate job matching and talent discovery to drive the career health scoring, position delta engine, and agent decision loops (such as matching a candidate profile to the best-fit opportunities in the market). This requires a highly performant and precise search infrastructure.

Search in this domain faces unique challenges:
1. **Keyword Rigidity**: Technical hiring requires matching exact terms (e.g., "FastAPI", "Go", "Kubernetes"). Standard semantic vector search often groups similar concepts but misses exact tool requirements (e.g., matching a "Go" query with a "Rust" or "Java" resume because both represent high-performance systems languages).
2. **Semantic Nuance**: Job descriptions and resume bullet points are filled with synonyms and varied phrasing (e.g., "high-throughput API design" vs "optimized REST endpoints handling millions of requests"). Standard keyword search (BM25) misses these conceptual overlaps.
3. **Agent Constraints**: LLM agents (like the Research Agent and Intelligence Agent) operate with finite context windows and high tokens-per-second costs. Feeding them raw, unranked, or low-quality job matches leads to poor synthesis, increased costs, and higher decision latency.

Per the guidelines in [IMPLEMENTATION_DOCTRINE.md](file:///C:/Users/shiva/Desktop/Projects/CareerPilot/docs/IMPLEMENTATION_DOCTRINE.md), single-stage retrieval is prohibited. The system must support a multi-stage search architecture that delivers highly precise matching within latency bounds (p95 < 400ms).

## Decision
We will implement a hybrid, multi-stage retrieval architecture for all search-related workloads. The pipeline is structured as follows:

```
┌────────────────────────────────────────────────────────────────────────┐
│                        RETRIEVAL PIPELINE STAGES                       │
│                                                                        │
│                          [Incoming Search Query]                       │
│                                     │                                  │
│                 ┌───────────────────┴───────────────────┐              │
│                 ▼ (Stage 1a)                            ▼ (Stage 1b)   │
│         [PostgreSQL BM25]                        [Qdrant Semantic]     │
│         Keyword search via                       Dense vector search   │
│         GIN English indexes                      (Bi-Encoder 1536d)    │
│                 │                                       │              │
│                 └───────────────────┬───────────────────┘              │
│                                     ▼ (Stage 2)                        │
│                           [Reciprocal Rank Fusion]                     │
│                           Merges top 100 from both                     │
│                           streams using constant k=60                  │
│                                     │                                  │
│                                     ▼ (Stage 3)                        │
│                           [Cross-Encoder Reranker]                     │
│                           Scores top 30 candidates                     │
│                           via deep token-attention                     │
│                                     │                                  │
│                                     ▼                                  │
│                            [Top 10 Sorted Results]                     │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

### Stage 1: Parallel First-Stage Retrieval (Recall-Optimized)
When a search query is received (or generated by an agent), the system executes two parallel searches to generate candidate pools:

#### A. Keyword Search (PostgreSQL BM25)
- **Mechanism**: PostgreSQL Full-Text Search (FTS) using `tsvector` columns on job postings and profile tables.
- **Index**: Generalized Inverted Index (`GIN`) built on English normalized configurations (`to_tsvector('english', title || description)`).
- **Output**: Returns the top 100 candidate IDs sorted by PostgreSQL `ts_rank`.

#### B. Dense Semantic Vector Search (Qdrant)
- **Mechanism**: Dense semantic vectors computed via an embedding model (e.g., OpenAI `text-embedding-3-small` or self-hosted `bge-small-en-v1.5` generating 1536-dimensional or 384-dimensional vectors).
- **Index**: Cosine similarity index in Qdrant Vector Database.
- **Output**: Returns the top 100 candidate IDs sorted by cosine score.

### Stage 2: Reciprocal Rank Fusion (RRF)
- **Mechanism**: The candidates from both streams are merged using the RRF algorithm. RRF calculates a new score for each candidate based on its relative rank in the two lists, rather than the raw scores (which are on incompatible scales).
- **Formula**:
  $$RRF\_Score(d \in D) = \sum_{m \in M} \frac{1}{k + r_m(d)}$$
  Where $M$ is the set of retrieval systems (Postgres and Qdrant), $r_m(d)$ is the rank of document $d$ in system $m$, and $k$ is a constant set to `60` (standard parameter empirically shown to balance keyword and semantic relevance).
- **Output**: Returns the top 30 fused candidates.

### Stage 3: Secondary Reranking (Precision-Optimized)
- **Mechanism**: The top 30 candidates from the RRF step are evaluated using a Cross-Encoder transformer model (e.g., `BAAI/bge-reranker-large` or the Cohere Rerank API).
- **Why**: Bi-Encoder models (used in Qdrant vector search) embed query and document independently, missing fine-grained interaction. Cross-Encoders feed both query and document text together into the attention layers, capturing token-to-token dependencies. This provides extremely high precision but is computationally expensive. Running it as a late-stage reranker on only 30 candidates makes it highly feasible within latency limits.
- **Output**: Returns the final top $N$ (e.g., 10) matched results to the calling agent or user dashboard.

### Data Synchronization
To ensure index consistency, an asynchronous Celery task executes every hour to identify unsynced PostgreSQL job postings, generate embeddings, and upsert them to Qdrant. Additionally, real-time inserts trigger immediate Qdrant upserts via event handlers.

## Alternatives Considered

### 1. Pure Semantic Vector Search (Single-Stage)
- **Description**: Relying entirely on dense vectors in Qdrant for search.
- **Why Rejected**: Vector embeddings compress long texts (like entire job descriptions) into single vectors. During this compression, specific hard requirements (like "Python v3.11+" or "Ashby API") get averaged out. As a result, search results contain conceptually close matches that lack critical technical prerequisites, which violates the strict requirements of career matching.

### 2. ElasticSearch / OpenSearch as Unified Store
- **Description**: Migrating job postings and profile search index to an Elasticsearch cluster to handle BM25, vector search, and hybrid scoring in one system.
- **Why Rejected**: Introducing Elasticsearch adds a third infrastructure database to maintain, increasing docker-compose and deployment complexity. For CareerPilot's MVP and V2 deployment, PostgreSQL and Qdrant are already present in the architecture. Using them in combination with an RRF Python service provides identical performance with zero additional infrastructure footprint.

### 3. ColBERT (Contextualized Late Interaction)
- **Description**: Using ColBERT (e.g., via Vespa or PLAID) for token-level interaction search.
- **Why Rejected**: Colbert is highly performant but requires specialized database indexes and custom library compilation, which is difficult to support on standard Python runtimes and CPU-only cloud VMs.

## Consequences

### Positive
- **Synonym & Exact Match Support**: Successfully retrieves "Kubernetes" when explicitly requested, while also fetching conceptual matches like "container orchestration engine platform engineer".
- **Enhanced Agent Output Quality**: Downstream agents receive clean, high-precision results, preventing context window bloat and decreasing LLM API costs.
- **Degraded Graceful Failure**: If Qdrant goes offline, the retrieval service automatically falls back to PostgreSQL BM25 keyword search, logging a warning but maintaining platform availability.
- **Scalability**: First-stage retrieval scales easily with database indexes, while the expensive Cross-Encoder only processes a fixed, small subset (K=30) of candidates.

### Negative
- **Increased Latency**: Running two parallel queries, merging them, and executing a Cross-Encoder step increases latency. We must optimize the Cross-Encoder execution (using ONNX Runtime, GPU instances, or API caching) to ensure p95 latencies remain under the 400ms limit.
- **Two Databases to Sync**: Developers must maintain PostgreSQL and Qdrant collections. If a sync job fails, vector search returns stale results. This requires robust sync failure monitoring.

## Tradeoffs

### Retrieval Quality vs. Execution Latency
We trade execution speed for search precision. A single-stage PostgreSQL search completes in < 15ms. The hybrid multi-stage pipeline takes 150-300ms. However, the high accuracy of matched jobs directly determines the quality of Career Health Scores and Position Delta explanations. High-quality results save users time and reduce downstream agent reasoning failures, making the latency cost highly worthwhile.

### Implementation Complexity vs. Infrastructure Footprint
We chose to implement RRF and Reranking logic in our FastAPI code layer using Postgres and Qdrant, rather than deploying Elasticsearch or Vespa. This keeps our database count low (avoiding operational complexity) but requires us to write custom candidate fusion and synchronization logic in our application repository.
